1. 一句话总结

创新点核心：标准 YOLO Detect head 是 3 尺度输出（P3/P4/P5 分别检测小/中/大目标）。Detect2 nl=2 把 P5 检测头删掉，只保留 P3+P4 两个尺度。

假设：钢卷头尾是固定大小的小目标（50×80 px），不需要大目标检测尺度，删除 P5 能减少参数 + 强制模型专注小目标。

本次结果：学术 mAP50 = 0.6332（vs baseline 0.8888，-25.6 pp），部署 F1 = 0.7407（vs v18.3 0.9286，-18.8 pp）。

失败根因：DFL 投影坍缩（最致命） P5 冗余互助缺失 数据规模硬约束 P5 路径半失效 TTA 不兼容

2. 背景知识：YOLO Detect Head 是什么

2.1 标准 YOLO 检测头的多尺度结构

YOLOv8 的 Detect head 在 3 个不同尺度的特征图上预测目标，对应不同大小的目标：

2.2 Detect head 的核心结构

每个检测尺度有 cv2（回归分支，4×reg_max=64 通道输出 DFL 距离）+ cv3（分类分支，nc=1 通道输出 sigmoid 概率）：

class Detect(nn.Module):
    def __init__(self, nc=80, ch=()):
        self.nc = nc                    # 类别数（钢卷场景=1）
        self.nl = len(ch)               # 检测层数（标准=3）
        self.reg_max = 16               # DFL 通道
        self.no = nc + reg_max * 4      # 每锚点输出维度（= 65）

        # 每个尺度独立一组 cv2/cv3
        self.cv2 = ModuleList(...)      # 回归: Conv → Conv → Conv2d(4*16)
        self.cv3 = ModuleList(...)      # 分类: Conv → Conv → Conv2d(nc)
        self.dfl = DFL(reg_max)         # 分布式焦点损失 → bbox

参数开销：每尺度约 250K 参数（3 个 Conv × ~83K）。3 尺度总开销 ~750K，占总模型的 19%（baseline 总参数 3.95M）。

3. 创新点本身：Detect2 nl=2 是什么

3.1 一句话定义

Detect2 nl=2 = 继承标准 Detect，但强制只检测 2 个尺度（P3 + P4），P5 检测头被删除。

3.2 与标准 Detect 的差异对比

4. 原理详解：为什么 P5 该删（架构假设）

核心假设

钢卷头尾目标 50×80 像素（在 2560×1440 原图上），对应在 1024×576 letterbox 后约 20×32 像素。

  • 在 P3 (stride=8) 特征图上：目标 = 2.5×4 像素 → 占 10 个 cell，可检出
  • 在 P4 (stride=16) 特征图上：目标 = 1.25×2 像素 → 占 2.5 个 cell，勉强可检出
  • 在 P5 (stride=32) 特征图上：目标 = 0.6×1 像素 → 占 0.6 个 cell，下采样过度，无法检出

结论：P5 对钢卷场景是冗余输出。删除 P5 检测头能减少参数 + 避免模型被"分散注意力"到不可检出的大目标。

4.1 架构假设的理论分析

出发点：钢卷头尾是 固定大小的小目标，场景单一（不变化尺度），所以 P5 大目标检测头是冗余。

三个潜在收益：

  1. 参数减少：1 个检测尺度 ≈ 250K 参数 → 总参数 -9.4%
  2. 推理加速：减少 1 个尺度的预测分支 → 略快（实测 12ms → 10ms）
  3. 训练聚焦：模型不被强制学习大目标模式（虽然实际没有大目标样本，但损失函数不会因 P5 空梯度浪费优化资源）

4.2 隐含的权衡

但实际上：

  • P5 输出在 neck 中仍生成（C2f 模块），但 Detect2 不消费 → 计算量并未真正减少
  • 检测头从 3 变 2 意味着回归分支和分类分支各自损失 1/3 的梯度信号 → 单尺度梯度信号减半
  • DFL 的 16 个 bin 分布在 P3+P4+P5 上训练时是 3×16=48 个有效 bin，nl=2 后只剩 2×16=32 个

5. 代码改动：Detect2 子类 + panet yaml

5.1 Detect2 子类（仅 14 行）

class Detect2(Detect):
    """YOLOv8 Detect head with only N3+N4 outputs (2 scales, P5 dropped).

    Inherits all forward logic and conv construction from Detect, but forces nl=2 by truncating
    the input channel tuple to its first 2 elements (N3, N4). Backbone + neck remain unchanged.
    """

    def __init__(self, nc=80, ch=()):
        """Initialize Detect2 with exactly 2 detection scales (N3, N4)."""
        ch2 = tuple(ch[:2])              # 只取前 2 个通道（N3, N4），P5 被截断
        super().__init__(nc, ch2)        # 复用 Detect 的所有构造逻辑
        self.nl = 2                      # 显式锁定 nl=2

7. 失败根因深度剖析（4 个层次）

7.1 根因 1：DFL 投影坍缩（最致命）

⚠ 用户反馈修正：初版报告说"Detect2 冷启动 / 只继承 2/3 预训练权重 / 损失 1/3 梯度信号"——这些说法大多不准确。真实机制是下面这套。

关键事实：Detect2 子类在 __init__ 里调用 super().__init__(nc, ch2)，父类 Detect 的构造会全新创建 cv2/cv3/dfl ModuleList（head.py line 37-40）。所以根本不存在"继承 2/3 预训练权重"——所有 750K head 参数都是随机初始化。

真正影响模型表现的不是 cv2/cv3 没学好，而是 self.dfl 这个整个 head 共享的模块（head.py line 40: self.dfl = DFL(self.reg_max) if self.reg_max > 1 else nn.Identity()）。

DFL 是什么

DFL = Distributed Focal Loss，把 bbox 边距的连续值离散化为 16 个 bin，每个 bin 配一个 Softmax 概率。最终预测距离 = Σ softmax(bin_i) × bin_i。

DFL 在 baseline vs Detect2 的差别

在 forward 里（head.py line 50），三个尺度（baseline）或两个尺度（Detect2）的 anchor 会被 make_anchors 拼起来：

# baseline (nl=3): anchors 来自 P3 高密度 + P4 中密度 + P5 稀疏  →  DFL 学到的距离分布很广
# Detect2 (nl=2): anchors 来自 P3 高密度 + P4 中密度 (P5 没了)   →  距离分布严重偏移

# forward line 53:
x_cat = torch.cat([xi.view(shape[0], self.no, -1) for xi in x], 2)
#           ↑
#    baseline: 3 尺度 anchor 拼接（共 h1*w1 + h2*w2 + h3*w3 个）
#    Detect2 : 2 尺度 anchor 拼接（共 h1*w1 + h2*w2 个，少 1/3 锚点）

# forward line 59:
dbox = dist2bbox(self.dfl(box), self.anchors.unsqueeze(0), xywh=True, dim=1) * self.strides
#                        ↑
#            DFL 对共享的 16 个 bin 做 Softmax 投影
#            baseline 训练的 DFL 已收敛在 3 尺度距离分布上
#            Detect2 推理时输入的 box 来自 2 尺度 → 分布错位 → 投影精度暴跌

为什么 DFL 投影坍缩

DFL 内部其实就是 1 个 Conv2d(reg_max*4, reg_max*4, 1)（head.py block.py 的 DFL 定义）。它的 16 个 bin 在训练时学到的是对应尺度 anchor 的距离分布峰值位置。

想象 DFL 是一个"分段直方图投影器"：

  • baseline 训练：D FL 看到 P3 距离 0-32px（密集 bin 0-8）+ P4 距离 0-64px（密集 bin 0-12）+ P5 距离 0-128px（密集 bin 4-16）。这 3 个分布共同把 16 个 bin 的梯度填满。
  • Detect2 训练：DFL 只看到 P3 距离 0-32px + P4 距离 0-64px。bin 13-16（对应大距离 96-128px）几乎不更新梯度 → DFL 在 bin 13-16 上的权重随机初始化值不再有意义。
  • Detect2 推理：bbox 真值的 4 条边距离在 0-32px 范围（钢卷场景 50×80 px 在 1024 输入上 stride=8/16），主要落在 bin 0-8——这部分分布还能用，但 bin 9-16 的"投影尾巴"成了噪声。

DFL Softmax 在分布外的 bin 上仍会输出非零概率（Softmax 永远不归零），这些噪声经过 Σ prob_i × bin_i 加权求和后会累积成系统性偏差——这才是 mAP50 暴跌 25pp 的真正机制。

为什么续训 + 121 ep 也无法突破 0.60

如果你看 panet 训练曲线，会发现 ep200 = 0.5894, ep250 = 0.5722, 续训到 ep121 ≈ 0.58。瓶颈不在 cv2/cv3 还在学，而在于 DFL 的 16-bin 分布假设被打破后，cv2 输出再准也救不回 Softmax 投影的系统性偏差。

这就是 Detect2 nl=2 "看着只删了一层、实际动了 DFL 这个不可拆分的耦合点" 的真实机制。

7.2 根因 2：P5 冗余互助缺失

现象：单尺度看 panet 每个尺度样本更多（~156 张 vs baseline ~104 张），但模型整体精度暴跌。

真正原因：3 尺度在 NWD 监督下是冗余互助的——同一目标在 P3/P4/P5 都被分配 anchor，三处梯度共同强化 bbox 定位。nl=2 后失去 P5 冗余信号，DFL 学到的位置分布更稀疏。

与"冷启动/梯度损失"的区别：这里不是说 cv2/cv3 没梯度（实际反而更多），而是 DFL 失去一个尺度的 anchor 分布作为正则化，分布稀疏导致 DFL 投影偏差放大。

7.3 根因 3：数据规模硬约束

现象：train=312 张正样本 + 233 负样本 = 545 张。

原因：Detect2 的 2 尺度设计相比 3 尺度：

项目3 尺度 (baseline)2 尺度 (Detect2)
每个尺度的有效训练样本~104 张/尺度~156 张/尺度
梯度信号维度3×16=48 bin2×16=32 bin (-33%)
DFL 距离预测样本量多尺度融合更稳定尺度少，DFL 学到的距离分布更稀疏

后果：在 545 张小数据集上，2 尺度的统计效能不足以支撑小目标检测的高精度需求。同 [[v17-p2-four-scale-fail]] 失败模式（4 尺度在 312 张数据上 mAP50=0.034 彻底失败）。

7.4 根因 4：P5 信息流"半失效"

现象：Detect2 删了 P5 检测头，但 backbone+neck 仍生成 N5。

原因：原 neck 中 N4 (idx 22) 接收来自 N5 (idx 20) 的 Concat 信息（line 30 [[-1, 20], 1, Concat]）。删除 P5 检测头不等于删除 N5 路径 —— N5 仍生成并参与 N4/N3 的特征融合，但梯度不再回传到 N5 检测。这造成 N5 路径"半失效"。

后果：N3/N4 失去了 P5 提供的语义信息补偿（虽然 P5 原本对小目标也没用，但 neck 中 N5→N4 Concat 提供了上下文特征）。

7.5 根因 5：TTA-builtin 不兼容

现象：用 model.predict(augment=True) 跑 TTA-builtin 时 RuntimeError：

RuntimeError: Sizes of tensors must match except in dimension 1.
Expected size 25 but got size 26 for tensor number 4 in the list.
File "ultralytics/nn/modules/conv.py", line 322, in forward
    return torch.cat(x, self.d)

原因：TTA-builtin 启用 multi-scale 推理（不同 imgsz 测试），但 Detect2 在某些尺度下 N3/N4 的 spatial size 不匹配 Concat 要求。Conv.py line 322 是 Proto 模块（用于 mask，但即使 nc=1 也包含）。

后果：部署时必须关 TTA (augment=False)，牺牲了 TTA 带来的精度提升。

8. 改进方向（5 个具体方案）

8.1 改进 A：反向 — 删 P3 而非 P5

思路：钢卷头尾是中等目标（20×32 px in 1024），实际更多落在 P4 而非 P3。删除 P3 (stride=8) 反而合理：

  • P4 (stride=16): 目标 = 1.25×2 像素 → 仍是 2.5 cell
  • P5 (stride=32): 目标 = 0.6×1 像素 → 0.6 cell（边缘）
  • 但 P3 + P4 + P5 三尺度在 NWD 监督下能互助，删除任一会损失

具体实现：再写一个 Detect_P4_P5 子类，保留 P4+P5。

预期收益：避免小目标信息缺失，保留大尺度辅助。

风险：P3 仍是最适合 20px 目标的尺度，删除会损失主要精度。

8.2 改进 B：保留 P5 但简化 head

思路：不是删 P5 检测头，而是共享权重让 P3 和 P5 用同一组 cv2/cv3：

class DetectShared(Detect):
    def __init__(self, nc=80, ch=()):
        super().__init__(nc, ch)
        # 共享 P3 和 P5 的 cv2/cv3（参数减半，3 尺度仍输出）
        self.cv2_p5 = self.cv2[0]    # 共享 P3 的权重
        self.cv3_p5 = self.cv3[0]

具体实现：在 cv2/cv3 ModuleList 长度=3 但 P5 索引指向 P3 权重。

预期收益：参数 -1/3，但保留 3 尺度信息流。

风险：P5 检测时实际用 P3 学到的特征，效果可能打折。

8.3 改进 C：From-Coil-PANet pretrain

思路：用 baseline best.pt 当 Detect2 的起点预训练权重，而不是从 random init：

PYTHONPATH= /home/pi/anaconda3/envs/hyper-yolo/bin/yolo detect train \
  model=runs/baseline/v0_baseline_hyper_yolon_strong_aug_250ep/weights/best.pt \

注意：Detect2 的 cv2/cv3 长度=2，baseline 的 cv2/cv3 长度=3，权重映射要写代码（继承前 2 个）。

预期收益：避免冷启动，节省 ~50 ep 的"消化"时间。

风险：load_state_dict 时 strict=False，丢弃 P5 索引权重。

8.4 改进 D：Detect2 + 弱 aug 范式

思路：v18.3 范式（弱 aug + HN crop + lr=0.005 + 100ep）证明对部署 F1 极有效。Detect2 也走此范式：

# run_detect2_weak_aug.sh

degrees=0.0 scale=0.0 flipud=0.0 copy_paste=0.0
# data 已含 hard neg 副本 (hn{1,2,3}_*.png)

预期收益：可能把部署 F1 从 0.74 提升到 0.85+ （参照 v0 baseline 强 aug 0.87 → v19 弱 aug 0.87 持平）。

8.5 改进 E：数据扩容（最关键）

思路：所有架构创新在 312 张正样本上都是数据规模受限。优先扩容：

  • 从 captures_update/ 目录提取更多 steel coil 时间序列
  • 用 scripts/expand_hard_neg.py + 半监督 pseudo-label 增加训练样本
  • 数据量目标 ≥1000 张正样本

预期收益：所有创新点的失败根因（数据规模硬约束）一次性解决。

这是根本性突破方向，比任何架构创新 ROI 都高。

9. 总结与展望

核心结论
